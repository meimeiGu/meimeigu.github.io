组件、实例和元素的不同让很多React新手感到迷惑，为什么会有三个不同的术语来指代在屏幕上绘制的东西。

原文：https://reactjs.org/blog/2015/12/18/react-components-elements-and-instances.html

##管理实例

如果你刚接触React,你可能之前只接触过组件类和实例。例如，你可能创造一个类来定义一个Button组件。当程序在运行的时候，你可能在屏幕上有多个此组件实例，每个实例都有自己的属性和状态。这是传统的面向对象的UI编程。为什么要介绍元素？

在传统的UI模型中，由你来负责创造或销毁子组件实例。如果一个Form组件渲染一个Button组件，它需要创造Button的实例，并手动使其与新信息保持同步。

这是伪代码，但是当你像用backbone这样的库用面向对象的方式写复合UI代码时，它或多或少是你最终的结果。

每个组件实例都必须保持对其DOM节点和子组件实例的引用，在合适的时机创建、更新和销毁它们。代码行的增长为组件可能状态数的平方，父代可以直接访问子组件实例，这使得将来很难将它们分离。

那么React不同在哪儿呢？

##元素描述树

在React中，这是elements要解决的地方。元素是用来描述组件实例或者DOM节点和它需要的属性的一个扁平对象。 它只包含了有关组件的类型（例如，Button）,它的属性（例如，颜色）和在组件里面的任何子元素。

元素并不是真实的实例。相反，它是一种告诉React你想在屏幕上看见什么的方法。你不能在元属上调用任何方法。它只是一个带有两个字段的不可变的描述对象：type:(string | ReactClass)和props:Object.

React元素很容易遍历，不需要解析，当然也比实际的DOM元素轻量级的多——它们只是对象！

这是调用reconciliation过程中的一部分，当你调用ReactDOM.render()或者setState()时启动。在协调结束的时候，React知道DOM树的结果，像react-dom或者react-native这样的渲染器应用会更新DOM节点所需的最小变更集（或者React Native的特定于平台的视图）

这种渐进的精炼过程也是React应用程序易于优化的原因。如果你的组件树有些部分变得太大导致React无法有效访问，你可以在相关props没有改变的情况下，区分树的某些部分，告诉React跳过这个“refining”。如果它们是不可变的，不管props是否改变计算都是非常快的，所以Reac和immutability可以很好地协同工作，并且可以用最小的付出提供最大的优化。

你可能注意到这篇博客讨论了很多关于组件和元素的内容，而对于实例不是很多。事实上，实例在React中的重要性远远低于大多数面向对象的框架。

只有组件定义为类拥有实例，并且你永远不能直接创建它们：React替你做了这些。虽然存在父组件实例访问子组件实例的机制，它们仅用于强制性操作（例如设置输入框焦点），并且通常应该避免。

React负责为每个类组件创建实例，所以你可以使用方法或者本地状态用面向对象的方式编写组件，但除此之外，实例在React编程模型中并不重要，并且由React自身管理。

总结

元素是一个普通对象，用来描述你希望在DOM节点或者其他组件方面在屏幕上显示的内容。元素可以在他们的props里包含其他元素。创造一个元素是很容易的，一旦元素创建了，它永远不会发生突变。

一个组件可以被几种不同的方式声明。它可以是一个类，有render()方法。或者，在简单的情况下，它可以被定义为一个函数。在任一情况下，组件获取props为输入，返回元素树作为其输出。

当一个组件接受一些参数作为输入，这是因为一个特定的父组件返回了一个有类型和这些参数的元素。这就是人们说props在React中流动的原因：从父组件到子组件。

实例就是你在编写组件类中所指的实例。它对于存储本地状态和执行生命周期事件非常有用。

函数式组件根本没有实例。类组件拥有实例，但是你永远不需要直接创建组件实例——React会负责这些。

最后，创建实例，可以用React.createElement(),JSX或者element factory helper.不要在真实的代码里用扁平对象来描写元素——只需要知道它们是外壳下的扁平对象。
